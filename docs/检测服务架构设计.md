# Detection-Service 架构设计 v2.0

> 纯 Worker 执行引擎。从 RocketMQ 消费消息，执行 HTTP 探测 + 匹配 + 提取，批量写结果。
> 不暴露 REST 接口，零外部 Feign 调用。

---

## 一、引擎分层架构

```
┌──────────────────────────────────────────────────────┐
│  TaskItemConsumer (RocketMQ Listener)                │
│  设置租户上下文，分发到 DetectionEngine               │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│  DetectionEngine (编排器)                             │
│  1. 解析 flow 表达式 → 步骤执行 DAG                    │
│  2. 逐步骤: 协议执行 → 提取变量 → 匹配判定             │
│  3. 汇总结果 → ResultWriter                          │
└──────┬───────────┬───────────┬───────────────────────┘
       │           │           │
┌──────▼──┐ ┌──────▼──┐ ┌──────▼──────────┐
│ Flow    │ │Protocol │ │ ResultWriter     │
│Interpr. │ │Registry │ │ 缓冲→批量写DB+Redis│
└─────────┘ └────┬─────┘ └──────────────────┘
                 │
    ┌────────────┼────────────┐
    │            │            │
┌───▼────┐ ┌─────▼────┐ ┌───▼─────┐
│ HTTP   │ │ TCP      │ │ DNS     │  ... (future)
│Handler │ │ Handler  │ │ Handler │
└───┬────┘ └──────────┘ └─────────┘
    │
    ├─ VariableContext (变量解析 + 占位符替换)
    ├─ MatcherPipeline (匹配器编排: and/or/negative)
    │    └─ MatcherRegistry (自动发现所有 Matcher Bean)
    └─ ExtractorPipeline (提取器编排)
         └─ ExtractorRegistry (自动发现所有 Extractor Bean)
```

**核心原则**：每种"能力"（匹配方式、提取方式、协议执行）都是可插拔的策略。新增一种能力 = 写一个 `@Component` 类，不改任何已有文件。

### 2.1 数据库实际用量

| 类型 | 记录数 | 模板数 | 占比 | 说明 |
|------|--------|--------|------|------|
| dsl | 67,257 | 3,155 | 29.5% | DSL 表达式匹配 |
| word | 12,891 | 7,283 | **68.2%** | 关键词包含 |
| status | 6,516 | 6,473 | **60.6%** | HTTP 状态码 |
| regex | 2,135 | 1,538 | 14.4% | 正则匹配 |
| binary | 8 | 8 | 0.1% | 十六进制匹配 |
| xpath | 2 | 2 | 0.0% | XPath 匹配 |
| size | 0 | 0 | 0% | （无模板使用） |

7,118 个模板（66.6%）使用了**多种 matcher 组合**——组合逻辑（and/or/negative）是核心能力。

### 2.2 策略接口

```java
/**
 * 匹配器策略。每种匹配类型一个 @Component 实现。
 * 策略自己解析自己的 config 字段，Pipeline 不关心 config 内部结构。
 */
public interface Matcher {
    /** 匹配器类型标识，对应 DB 中 vul_matcher.type 字段 */
    String type();

    /**
     * 对 HTTP 响应执行匹配。
     * @param ctx   HTTP 响应上下文（状态码/body/headers）
     * @param def   匹配器定义（公共字段 + type-specific config Map）
     * @return true 如果匹配成功
     */
    boolean match(HttpResponseContext ctx, MatcherDef def);
}
```

### 2.3 MatcherDef —— 瘦 DTO

```java
@Data
public class MatcherDef {
    private String type;              // "word" / "status" / "dsl" / "regex" / ...
    private String part;              // body / header / all / body_2 / content_type / ...
    private String condition;         // inner and/or (多条规则之间)
    private boolean negative;         // 是否取反
    private boolean caseInsensitive;  // 忽略大小写
    private Map<String, Object> config;  // 类型专属配置，各策略自己解析
}
```

**为什么 config 用 Map？** 数据链路是 `DB JSON → Feign JSON → MQ JSON`，全程序列化。Map 直通，零转换。每个 Matcher 内部解析自己的 config（如 WordMatcher 取 `config.get("words")`），脏活隔离在策略内部。

### 2.4 注册表 —— Spring 自动发现

```java
@Component
public class MatcherRegistry {
    private final Map<String, Matcher> matchers;

    /** Spring 自动注入所有 Matcher 实现 */
    public MatcherRegistry(List<Matcher> matcherList) {
        this.matchers = matcherList.stream()
            .collect(Collectors.toMap(Matcher::type, Function.identity()));
    }

    public Matcher get(String type) {
        Matcher m = matchers.get(type);
        if (m == null) throw new UnsupportedMatcherTypeException(type);
        return m;
    }
}
```

**新增匹配器 = 写一个类，加 `@Component`，搞定。**

### 2.5 编排器 —— MatcherPipeline

```java
@Component
public class MatcherPipeline {
    private final MatcherRegistry registry;

    /**
     * 对一组 matcher 执行组合判定。
     * @param ctx             HTTP 响应
     * @param defs            匹配器定义列表
     * @param outerCondition  外层 and/or（步骤级 matchers-condition）
     * @return true 如果整体匹配
     */
    public boolean evaluate(HttpResponseContext ctx,
                            List<MatcherDef> defs,
                            String outerCondition) {
        if (defs == null || defs.isEmpty()) return false;

        Predicate<MatcherDef> evaluator = def -> {
            Matcher matcher = registry.get(def.getType());
            boolean result = matcher.match(ctx, def);
            return def.isNegative() != result;  // XOR
        };

        return "and".equals(outerCondition)
            ? defs.stream().allMatch(evaluator)
            : defs.stream().anyMatch(evaluator);
    }
}
```

Pipeline 的职责只有：**遍历 + and/or 编排 + negative 取反**。不解析 config，不做类型分发。

### 2.6 具体策略示例

```java
@Component
public class WordMatcher implements Matcher {
    @Override public String type() { return "word"; }

    @Override
    public boolean match(HttpResponseContext ctx, MatcherDef def) {
        List<String> words = toStrList(def.getConfig(), "words");
        if (words == null || words.isEmpty()) return false;

        String target = resolvePart(ctx, def.getPart());

        return evaluateInner(words, def.getCondition(), word -> {
            if (def.isCaseInsensitive()) {
                return target.toLowerCase().contains(word.toLowerCase());
            }
            return target.contains(word);
        });
    }

    // resolvePart: body → ctx.getBody(), header → ctx.getHeaders().toString(),
    //              all → body + headers, body_2 → ctx.getPreviousBody(2), ...
}
```

```java
@Component
public class RegexMatcher implements Matcher {
    @Override public String type() { return "regex"; }

    @Override
    public boolean match(HttpResponseContext ctx, MatcherDef def) {
        List<String> patterns = toStrList(def.getConfig(), "regex");
        if (patterns == null || patterns.isEmpty()) return false;

        String target = resolvePart(ctx, def.getPart());

        return evaluateInner(patterns, def.getCondition(), pattern -> {
            return Pattern.compile(pattern, Pattern.DOTALL).matcher(target).find();
        });
    }
}
```

每个策略内部共享 `evaluateInner(List<T> rules, String condition, Predicate<T> tester)` —— 放在抽象基类中，处理 and/or 逻辑。

### 3.2 策略接口

```java
public interface Extractor {
    String type();

    /**
     * 从 HTTP 响应中提取变量值。
     * @param ctx  HTTP 响应上下文
     * @param def  提取器定义
     * @return 提取到的值，null 表示未提取到
     */
    String extract(HttpResponseContext ctx, ExtractorDef def);
}
```

### 3.3 ExtractorPipeline（与 MatcherPipeline 镜像设计）

```java
@Component
public class ExtractorPipeline {
    private final ExtractorRegistry registry;

    public void extract(HttpResponseContext ctx,
                        List<ExtractorDef> defs,
                        VariableContext vars) {
        if (defs == null) return;
        for (ExtractorDef def : defs) {
            Extractor extractor = registry.get(def.getType());
            String value = extractor.extract(ctx, def);
            if (value != null && def.getName() != null) {
                vars.set(def.getName(), value);
            }
        }
    }
}
```

## 四、协议抽象

### 4.1 接口

```java
/**
 * 协议处理器 — 执行一种协议（HTTP/TCP/DNS/SSL）的探测请求。
 * 每种协议一个 @Component 实现，由 ProtocolRegistry 自动收集。
 */
public interface ProtocolHandler {
    /** 协议名，对应模板中协议字段：http / tcp / dns / ssl */
    String protocol();

    /** 执行探测，返回协议无关的响应上下文 */

}
```

### 4.2 HTTP 实现（主力）

当前阶段只实现 `HttpProtocolHandler`。接口先抽出来，成本极低（一个 interface + 一个实现），但为 TCP/DNS 等扩展留好了位置。

### 4.3 HTTP 客户端升级要点

| 能力 | 当前状态 | 目标 |
|------|---------|------|

## 五、DSL 引擎 —— MVEL → Aviator

### 5.1 当前问题

DslMatcher 有一个 `preprocess()` 方法做"语法扭曲"：

```java
// MVEL 中 contains/regex 是操作符关键字，不能当函数名用
// 所以必须做这种转换：
contains(body, 'keyword')   →  body.contains('keyword')    // 语义扭曲
regex('pattern', body)      →  _regex('pattern', body)      // 绕道静态方法
```

这是 MVEL 不适合做 DSL Host 的铁证。

### 5.2 Aviator 优势

| 维度 | MVEL 2.x | Aviator |
|------|----------|---------|
| 维护状态 | 2017 年停止 | **活跃维护（2025）** |
| 性能 | 解释执行为主 | **编译为字节码，快 5-10x** |
| 安全性 | 可执行任意 Java 代码，**有已知 CVE** | **沙箱模式**，默认禁反射/IO |
| 自定义函数 | ParserContext + 反射 | `addFunction(new AbstractFunction(){})` |
| 语法冲突 | contains/regex 是关键字 | **无冲突，直接当函数名用** |
| 体积 | ~1.5MB | **~500KB** |

### 5.3 换 Aviator 后的 DSL 函数注册

```java
@Component
public class DslEngine {
    private final Map<String, Object> builtinFunctions = new HashMap<>();

    public DslEngine() {
        AviatorEvaluator.addFunction(new ContainsFunction());
        AviatorEvaluator.addFunction(new ContainsAllFunction());
        AviatorEvaluator.addFunction(new ContainsAnyFunction());
        AviatorEvaluator.addFunction(new RegexFunction());
        AviatorEvaluator.addFunction(new LenFunction());
        AviatorEvaluator.addFunction(new ToLowerFunction());
        AviatorEvaluator.addFunction(new ToUpperFunction());
        AviatorEvaluator.addFunction(new CompareVersionsFunction());
        AviatorEvaluator.addFunction(new ConcatFunction());
        AviatorEvaluator.addFunction(new HexEncodeFunction());
        AviatorEvaluator.addFunction(new UrlDecodeFunction());
        // ... 按需添加
    }

    public boolean evaluate(String expression, Map<String, Object> env) {
        Expression compiled = AviatorEvaluator.compile(expression, true);
        Object result = compiled.execute(env);
        return result instanceof Boolean b ? b
             : result instanceof Number n ? n.intValue() != 0
             : false;
    }
}
```

模板中的 DSL 表达式如 `contains(body, 'admin') && status_code == 200` 直接原生执行，不需要 `preprocess()`。

## 六、多步骤响应引用（body_2 / header_3 等）

数据库统计显示 `vul_matcher.part` 有 30+ 种值，包括 `body_2`、`header_3`、`content_type_2` 等。这些是多步骤模板中引用**前面步骤的响应**。

### 6.1 设计

`HttpResponseContext` 需要持有步骤间共享的**历史响应注册表**：

```java
public class HttpResponseContext {
    // 当前步骤响应
    private int statusCode;
    private String body;
    private Map<String, List<String>> headers;
    private String contentType;
    private int contentLength;
    private long durationMs;

    // 多步骤历史（step order → 响应快照）
    private Map<Integer, StepResponseSnapshot> history;
}
```

`resolvePart()` 逻辑：
- `body` → 当前步骤 body
- `body_2` → `history.get(2).getBody()`
- `header_3` → `history.get(3).getHeaders().toString()`
- `all` → 当前 body + headers 拼接

## 七、Flow 执行引擎

Nuclei 模板的 `flow` 字段定义多步骤执行顺序。当前实现只支持简单的 `http(N)` 顺序执行。

### 7.1 接口抽象

```java
public interface FlowInterpreter {
    /**
     * 按 flow 表达式执行步骤序列。
     * @param steps    所有步骤
     * @param flow     flow 表达式（null/空 = 单步）
     * @param executor 单步执行器
     * @return true 如果整体匹配
     */
    boolean execute(List<StepDef> steps, String flow, StepExecutor executor);
}
```

### 7.2 实现层次

| 级别 | 能力 | 实现方式 |
|------|------|---------|
| V1（当前） | 无 flow / 顺序 | DefaultFlowInterpreter |
| V2 | `http(1) && http(2)` / `http(1) \|\| http(2)` | 简单布尔表达式解析 |
| V3 | `for x in vars { http(1) }` | 需完整 DSL 解析器 |

当前聚焦 V1，V2/V3 的接口先定义好。

## 八、VariableContext（变量上下文）

### 8.1 变量优先级

```
Extractor 提取变量  >  模板级 variables  >  Asset 占位符  >  随机生成函数
      (运行时)          (模板定义)          (BaseURL等)      (randstr等)
```

### 8.2 设计

```java
public class VariableContext {
    private final Map<String, Object> layers = new HashMap<>();

    // → 从消息构建
    public static VariableContext from(TaskItemMessage msg) { ... }

    // → Extractor 写入（高优先级）
    public void set(String key, Object value) { layers.put(key, value); }

    // → 占位符替换 {{...}}
    public String resolve(String template) { ... }

    // → 响应变量同步（body, status_code, content_type 等）
    public void updateFrom(ProtocolResponse response) { ... }
}
```

不再用 `get()` + `put()` 暴露内部 Map，而是提供语义化的 `resolve()` 和 `set()`。

## 十、改造后的 DetectionEngine

```java
@Component
public class DetectionEngine {
    private final ProtocolRegistry protocolRegistry;
    private final MatcherPipeline matcherPipeline;
    private final ExtractorPipeline extractorPipeline;
    private final FlowInterpreter flowInterpreter;
    private final ResultWriter resultWriter;

    public void execute(TaskItemMessage msg) {
        VariableContext vars = VariableContext.from(msg);

        try {
            List<StepDef> steps = StepDefMapper.from(msg);

            boolean matched = flowInterpreter.execute(steps, msg.getFlow(),
                step -> executeStep(step, msg.getAssetProtocol(), vars));

            writeResult(msg, matched ? "matched" : "not_matched", vars);
        } catch (Exception e) {
            writeError(msg, e);
        }
    }

    private boolean executeStep(StepDef step, String protocol, VariableContext vars) {
        // 1. 协议执行
        ProtocolHandler handler = protocolRegistry.get(protocol);
        ProtocolResponse response = handler.execute(step, vars);

        // 2. 同步响应变量到上下文
        vars.updateFrom(response);

        // 3. 提取（必须在匹配之前 — 变量可能被后续步骤的 matcher 引用）
        extractorPipeline.extract(response, step.getExtractors(), vars);

        // 4. 匹配
        if (step.getMatchers() != null && !step.getMatchers().isEmpty()) {
            return matcherPipeline.evaluate(response, step.getMatchers(),
                   step.getMatchersCondition());
        }
        return false;
    }
}
```

**对比当前代码：**
- `toStrList` / `toIntList` / DTO→MatcherConfig 映射 → 统一收到 `StepDefMapper` 中
- 硬编码的策略/分发 → 全部走 Registry
- 引擎职责：编排 + 调度。不解析 config，不做类型判断。
