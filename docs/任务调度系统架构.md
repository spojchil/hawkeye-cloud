# 任务调度系统架构设计

## 概述

任务调度系统是 Hawkeye Cloud 的**核心检测引擎**，负责将用户提交的漏洞检测任务分解、分发到多个工作节点并行执行。采用 **三层队列架构**（预处理 → 工作 → 死信），基于 RocketMQ 实现异步解耦和可靠投递。

> **对应微服务：** 本架构由三个核心服务落地——**`task-service`**（预处理队列）、**`detection-service`**（工作队列、多实例 Worker）、死信队列由 detection-service 内建的补偿逻辑 + RocketMQ 原生死信机制共同处理。漏洞模板数据由 **`vul-service`** 独立管理，资产数据由 **`asset-service`** 提供。详细服务说明见 [模块说明](模块说明.md)。

## 整体架构

```
                   ┌──────────────────────────────────┐
                   │         客户端提交任务              │
                   └──────────────┬───────────────────┘
                                  │
                                  ▼
              ┌──────────────────────────────────┐
              │      预处理队列 (Topic)           │
              │   task_preprocess_topic          │
              │  职责：校验 + 拆分 + 智能分发       │
              └──────────────┬───────────────────┘
                             │
              ┌──────────────┼──────────────────┐
              │              │                    │
              ▼              ▼                    ▼
      ┌─────────────┐ ┌─────────────┐    ┌─────────────┐
      │  Worker 1   │ │  Worker 2   │    │  Worker N   │
      │ (MessageQueue)│ │ (MessageQueue)│  │ (MessageQueue)│
      │ 线程池并发执行 │ │ 线程池并发执行 │  │ 线程池并发执行 │
      └──────┬──────┘ └──────┬──────┘    └──────┬──────┘
             │               │                   │
             │    失败(重试耗尽)                    │
             └───────────────┼───────────────────┘
                             │
                             ▼
              ┌──────────────────────────────────┐
              │       死信队列 (%DLQ%)            │
              │   补偿服务消费 + 异常诊断           │
              └──────────────────────────────────┘
```

## 1. 任务预处理队列

### 核心职责

| 职责      | 说明                                                                                                                    |
|:--------|:----------------------------------------------------------------------------------------------------------------------|
| 任务异步解耦  | 对外提供任务提交入口，接收原始任务信息后立即返回响应，任务消息写入 RocketMQ Topic（`task_preprocess_topic`），由独立的拆分服务异步消费处理                              |
| 数据完整性校验 | 拆分服务消费任务消息后，校验资产 URL 格式、关联 POC 配置完整性，避免无效或错误的组合进入下游流程                                                                 |
| 智能分发    | 将校验通过的任务拆解为独立的检测组合，通过 RocketMQ 的 `MessageQueueSelector` 接口，结合 Redis 中存储的工作节点实时负载信息，定向发送到当前负载最低的工作节点所对应的 Message Queue |
| 状态管理    | 统一管理任务生命周期，提供从提交、拆分、分发、执行到完成的全链路状态追踪                                                                                  |
| 容错保障    | 实时检测工作节点异常，自动触发任务重新分发；重启工作节点可触发 RocketMQ 的 Rebalance 机制，实现队列重新分配                                                      |

### 业务场景示例

假设某企业用户提交了一个包含 **500 个检测组合** 的任务：

1. **任务接收** — 调度服务将任务元信息持久化至数据库，发送一条任务消息到 `task_preprocess_topic`
2. **校验与拆分** — 拆分服务拉取该任务消息，对 500 个资产的 URL 及其关联 POC 进行完整性校验（约 3 秒）
3. **负载感知分发** — 拆分服务生成 500 条独立的组合消息，实时查询 Redis 中 8 个工作节点的负载情况
4. **定向投递** — 根据各节点负载权重，通过 `MessageQueueSelector` 将组合消息分别发送至 8 个工作节点对应的 Message Queue
5. **故障转移** — 若某工作节点宕机，RocketMQ 的 Rebalance 机制自动将其队列重新分配给其他存活 Worker

## 2. 工作队列（Worker Queue）

### 核心职责

| 职责      | 说明                                                                                    |
|:--------|:--------------------------------------------------------------------------------------|
| 多线程并发优化 | 工作节点消费到组合消息后，提交到内部线程池并发执行。50 个组合串行 15s → 16 线程并发约 1s，提升约 15 倍                         |
| 队列分配    | RocketMQ 自动采用默认均衡分配策略，32 个 Message Queue 平均分配给 8 个 Worker（每实例 4 个队列），实例增减自动 Rebalance |
| 负载上报    | 每个 Worker 以固定间隔（如 5s）向 Redis 上报实时状态，为智能分发提供数据支撑                                       |
| 容错机制    | HTTP 请求超时（如 10s）、连接池复用；失败消息利用 RocketMQ 重试机制，重试耗尽后进入死信队列                               |

### 负载上报数据结构

```json
{
  "workerId": "worker-1",
  "queueSize": 23,
  "activeThreads": 12,
  "maxThreads": 32,
  "lastReportTime": 1716875300000
}
```

## 3. 死信队列（Dead Letter Queue）

### 核心职责

| 职责      | 说明                                                                                                      |
|:--------|:--------------------------------------------------------------------------------------------------------|
| 保证任务完整性 | RocketMQ 原生死信机制，失败消息自动进入独立死信 Topic（`%DLQ%<ConsumerGroup>`），由补偿服务统一消费。约 95% 因网络超时等瞬时故障失败的任务，在补偿服务中重试后可成功 |
| 异常问题诊断  | 补偿服务记录详尽异常上下文（请求参数、响应内容、错误堆栈），支持按异常类型、时间等维度统计分析，助力 POC 维护人员定位深层问题                                       |
| 系统稳定性保障 | 异常与正常任务物理隔离（独立 Topic），补偿服务以批量、低优先级方式处理，避免问题任务冲击核心检测链路                                                   |

## 技术选型

| 组件   | 技术                    | 说明                        |
|------|-----------------------|---------------------------|
| 消息队列 | Apache RocketMQ 5.3.3 | 高吞吐、低延迟、支持消息回溯和死信         |
| 负载感知 | Redis                 | 存储 Worker 实时负载信息，用于智能分发决策 |
| 分布式锁 | Redisson              | 保障任务分发等场景的原子性             |
| 工作节点 | Spring Boot 独立服务      | 内部线程池并发执行检测               |

## 实施路线

| 阶段         | 内容                                                        |
|:-----------|:----------------------------------------------------------|
| 1. 基础设施    | RocketMQ 环境搭建、Topic/ConsumerGroup 配置、Producer/Consumer 集成 |
| 2. 预处理队列   | 拆分服务实现：任务消息消费、校验、拆分、状态管理                                  |
| 3. 工作队列    | Worker 节点实现：组合消息消费、线程池管理、负载上报                             |
| 4. 智能分发    | 负载采集 + MessageQueueSelector 实现负载感知分发                      |
| 5. 死信 + 补偿 | 死信消费、补偿重试、异常统计、监控告警                                       |
