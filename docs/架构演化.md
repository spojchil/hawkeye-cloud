## 一、v1 早期设计 (Apr 22-26)

> 项目蓝图和架构设计阶段。

### 1.1 8 服务蓝图 → 6 服务落地

最初勾画了 8 个微服务的完整架构：

| 服务 | 蓝图 | 实际 |
|------|------|------|
| gateway-service | ✅ | ✅ 基础 |
| auth-service | ✅ | ✅ 基础 |
| asset-service | ✅ | ✅ |
| vul-service | 📝 | ✅ |
| task-service | 📝 | ✅ |
| detection-service | 📝 | ✅ |
| **tenant-service** | 📝 | **❌ 延后** |
| **audit-service** | 📝 | **❌ 延后** |

经过权衡，多租户在 MyBatis-Plus `TenantLineHandler` 层面已实现 SQL 级隔离，独立租户服务和审计服务在 MVP 阶段优先级不够，延后到 v2。

### 1.2 三层队列调度 → 单 Topic + 线程池

蓝图阶段的"预处理队列 → 工作队列 → 死信队列"涉及 3 个 RocketMQ Topic。实际编码中简化：

```
蓝图:  预处理 Topic → 工作 Topic → 死信 Topic （3 个 MQ 环节）
实现:  单个 task_item_topic，拆分在 @Async 线程池完成
```

拆分逻辑计算量不大（主要是 Feign + 笛卡尔积），不需要单独 MQ 环节。死信直接用 RocketMQ 原生 `%DLQ%`。

### 1.3 检测匹配策略：策略模式 → 注册表 + 管线

蓝图阶段设想过按漏洞类型分策略：
```
DetectionStrategy → SqlInjectionStrategy / XssStrategy / SsrStrategy ...
```
很快意识到上千种漏洞类型无法穷举，与 AI 讨论后改为按匹配方式分（Word/Status/Dsl/Regex），通过注册表 + 管线组合任意检测逻辑。

### 1.4 Python 导入 → Java 级联入库

蓝图阶段设计了 Python 脚本做模板导入，实现阶段改为 Java，与微服务集成更紧密。

## 二、vul-service 首次实现 (feature/vul-service, Apr 28-29)

### 2.1 SQL vs 文件存储——数据库优先

| 方案 | 优点 | 缺点 |
|------|------|------|
| 文件系统 | 无需建表 | 无法搜索、无分类、无版本管理 |
| MySQL | 可搜索/分类/版本/租户隔离 | 需建表+导入脚本 |

决定入库。10,688 个模板导入 MySQL 后通过 LIKE 做关键词搜索，树形分类做浏览。

### 2.2 实体模型——MVP 用 JSON，v3 才归一化

| 阶段 | 方案 |
|------|------|
| v1 MVP | 单表 3 个 JSON 字段存 http_requests/matchers/extractors |
| v3 | 拆 12 张归一化表 |

MVP 用 JSON 存储——模板嵌套结构复杂，拆表会导致大量联表查询，违反"禁止联表"原则。v3 阶段参考 AI 分析后认为归一化利大于弊，执行了全量重构。

> v1 的 JSON 决策在 v3 被反转——不是朝令夕改，而是 MVP 敏捷和成熟期归一的合理阶段切换。

### 2.3 标签 vs 分类——两条线并行

YAML 的 tags 是扁平标签（`cve,cve2024,paloalto`，7,255 个），与树形 `vul_category` 语义完全不同。有讨论将 tags 直接映射到 category，最终决定两条线并行，互不合并。

### 2.4 matchers JSON Schema——两层条件的接口契约

Nuclei 模板有两层条件逻辑：
```yaml
matchers-condition: and      # 外层：matcher 间关系
matchers:
  - type: word, condition: and, words: [...]  # 内层：matcher 内规则关系
```

JSON 存储时保留两层条件。这个 schema 成为 vul-service 和 detection-service 的接口契约。

### 2.5 其他实际问题

| 问题 | 修复 |
|------|------|
| VulSeverityEnum 类型矛盾（IEnum 但实体是 String） | 去掉 IEnum<Integer> |
| 实体遗漏 flow/variables 字段 | 补充，多步模板必需 |
| N+1 查重（逐条 selectOne 10,000+ 次） | 批量 IN 查询 |
| MetaObjectHandler 未扫描 | 新建 AdditionalScanConfig |
| Jackson 不可用 | 改用 fastjson2 |
| FULLTEXT 索引误用 | 删除索引，保留 LIKE |

## 三、task-service 初始实现 (feature/task-service, Apr 29)

### 3.1 实现策略：对标 asset-service

四层模块结构、pom 依赖、枚举实现 IEnum、MapStruct 转换——完全对标 asset-service 风格，不做任何创新。

### 3.2 设计文档矛盾

早期在 ddocs/ 下积累的多份设计文档存在多处不一致，逐条比对后决定：

| 冲突 | 决策 |
|------|------|
| 预处理 Topic 名 vs 单 Topic | 以单 Topic 为准 |
| 端口未明确 | 修正为 8005 |
| task_item 回写方式 | Redis 计数方案 |
| 消息体大小（仅 ID vs 全量） | 先保持仅 ID，后续改为全量 |

### 3.3 踩坑记录

| 问题 | 排查和修复 |
|------|-----------|
| RocketMQ Starter 版本 | 父 POM 误用服务端版本 5.3.3，显式声明 starter 2.3.2 |
| Redisson 3.32.0 不兼容 SB 4.0.5 | `RedisAutoConfiguration` 类已改名，移除依赖 |
| RedisTemplate 类型 | Spring Boot 默认只创建 `String,String`，改用 String 后 parseInt |
| @Async 不生效 | 自调用问题 + 线程池未定义，拆出独立 TaskSplitService |

### 3.4 代码审查——对标修复

对比 asset-service 发现 6 个问题：`getById` 绕路改为直查、cancel TOCTOU 改为原子更新、手写 VO 改 MapStruct、Scheduler 改注入 Service 等。AI 补充了 `cancel` 的原子更新建议。

## 四、v2 重构——检测引擎引入 (Apr 30 - May 1)

### 4.1 detection-service 的设计哲学

三个核心决策：
- **纯 Worker 无 API**：不暴露 REST，仅作 RocketMQ Consumer
- **消息全量携带**：省去 Feign 调用，detection 零外部依赖
- **可水平扩展**：无状态 + 多实例 + RocketMQ Rebalance

"消息体大小"的争议（仅传 ID vs 全量携带）经过讨论后走向全量。

### 4.2 匹配器注册表 + 管线——分 Phase 协作实现

v1 就确定了匹配架构的方向（注册表 + 管线替代策略模式）。v2 中分 Phase 实现：

- Phase 1：DDL 对齐 + 消息模板
- Phase 2：Matcher 注册表体系（接口 + Registry + Pipeline）
- Phase 3：Extractor 注册表体系
- Phase 4-5：Engine 瘦身 + VariableContext + FlowInterpreter + ResultWriter

提交：`8b9a819 → 8915f64 → 9f6262b`

### 4.3 负载感知分发——设计但主动砍掉

v2 设计了 Worker 定时上报 Redis 负载、MessageQueueSelector 定向投递的方案，但实现阶段主动砍掉——优先走通核心链路，负载感知属于优化项。`WorkerLoad.java` 后来被确认为死代码并删除。

## 五、v3 归一化与重构 (May 1-3)

### 5.1～5.3 数据库改进

- **12 表归一化**：参考 AI 分析后从 JSON 存储迁移到 12 表
- **deleted → deleted_at**：时间戳在复合唯一键中有优势
- **BIGINT → VARCHAR(64)**：直接存用户名，避免 JOIN

### 5.4 MVEL → Aviator——AI 发现安全风险，共同决定迁移

AI 在审查 DslMatcher 时发现 MVEL 的 RCE 风险（可调用 `Runtime.getRuntime().exec()`）。判断为高危安全问题，决定迁移到 Aviator。

提交：`acff1ec`

### 5.5 VariableResolver → VariableContext

提出"需要跨步骤变量回写"的需求后，设计了 Variables 优先级（提取器 > 模板 > 内置）、嵌套解析（5 层上限）的 VariableContext 体系。

提交：`db7cff6`、`9f6262b`

### 5.6 TaskItem 与 DetectionResult 合并

两张表存了相同信息（status、result 结果等），决定合并为一张表。

提交：`178db77`

### 5.7 枚举修正

AI 发现 `TaskStatusEnum` 比 DDL 多了一个 `PROCESSING` 状态，以 DDL 为准删除。

提交：`a7ddd1c`

## 六、AI 审查期——study-audit 分支 (May 4-6)

### 6.1 分支策略

```
main ← develop ← study-audit
```

AI 辅助开发的改动在 study-audit 分支进行，审查确认后合并到 develop。这样 develop 始终是稳定可构建的状态。

### 6.2 多会话上下文断裂

项目经历了多个 AI 会话流转。每次新会话完全不知道前序决策——不理解为何某设计被否决，重复分析已审查的模块，可能推翻前序的合理建议。建立 `架构演化.md` 作为会话间上下文传递的媒介。

### 6.3 四轮审查协作

**第一轮** (`6bd121c`，`591e0cf`)：
- 逐模块审查，发现 35 个问题（4 个 P0 + 6 个 P1 + 6 个 P2 + 5 个 P3）
- 删除 TaskInternalController（detection 已直写，Feign 回写成为死代码）
- 拆分 `splitAndDispatch` 大方法（150+ 行）
- 日志级别调整（debug→warn/info）

**第二轮**——关键 bug 修复：
- Redis 键不匹配：`ResultWriter` 写的键名和 `TaskProgressScheduler` 读的不同，导致任务永不标记 DONE
- HTTP 相关问题：URI 编码错误、HTTPS 降级、空 path 处理

**第三轮**——缓存与幂等：
- TemplateCache 三级防护：空值哨兵 + 逻辑过期 + 随机 TTL
- 模板 vs 资产缓存取舍：讨论后决定模板缓存、资产不缓存（资产变更频繁、实时性要求高）
- 幂等性审查：梳理全链路后发现 3 个缺口，设计了三重保护方案

**第四轮**——设计讨论：
- 资产标签 vs 检测匹配：多轮讨论后，决定"不能依赖自动识别，用户手动指定更可靠"。两套标签体系已设计但先撤回。
- 模板规模分析：AI 统计出 10,705 个模板的分布，指出 7,255 个标签过于扁平、分类数据未被使用，需要重新设计面向客户的分类体系。

### 6.4 讨论中的分歧与取舍

以下几个决策经过了反复讨论：
- **资产缓存**：AI 建议资产也加缓存，但资产变更频繁、实时性高，最终决定不缓存
- **预检器拆分**：有建议把预检器再拆成多个类，讨论后认为过度工程化，保持现状
- **消息体大小**：先保持仅传 ID，v2 阶段重新讨论后改为全量携带——说明技术决策需要随着上下文演进

## 七、收尾工作 (May 4-6)

### 7.1 Docker Compose 完善

发现 `docker compose up` 会尝试拉取 `ghcr.io` 远程镜像导致本地失败，排查出 `build:` + `image:` 并存的坑，决定微服务去掉 image 行。补充了 task/detection 的 Dockerfile，加入 RocketMQ，将 node.js 前端替换为 python:3-alpine。

### 7.2 文档整理

提出"文档混乱、老的新混在一起"后，协作完成了：API 和 SQL 迁移到 `docs/api/` 和 `docs/sql/`、7 个独立模块文档替代单一文件、删除过时文档 11 篇、README 重写为简洁开源风格。

### 7.3 develop → main 合并

试合并发现 18 个冲突文件（main 上旧 PR #7 的旧代码 vs study-audit 的 v3 重构代码）。以 develop 为准（`-X ours`），手动解决一个 modify/delete 冲突，合并后删除 main 带入的 11 个旧文件。

### 7.4 负载感知描述修正

比对老文档和实际代码时发现简历中错误写了"负载感知投递"，立即修正简历和面试要点，同时删除 `WorkerLoad.java` 死代码。

## 八、设计决策汇总

| 决策 | 从 | 到 | 原因 |
|------|-----|-----|------|
| 服务数量 | 8 | 6 | tenant/audit 优先级低 |

## 九、人机协作反思

### 9.1 共同决策模式

项目经历了多个 AI 会话。大部分决策是共同讨论的结果——有人先发现问题，双方反复推敲方案，最终达成一致。少数场景确实存在分歧（如资产缓存、预检器拆分），这是正常的协作过程。

### 9.2 提交历史中的协作

```
6bd121c  chore(task): 初步审查           ← 共同审查
591e0cf  refactor(task): 审查修复        ← 讨论 + 深度重构
0f808fb  feat: 增加幂等性保护            ← 发起审查 + 方案讨论 + 实施
a5f6a6e  docs: 全面整理项目文档结构      ← 提出需求 + 执行
823c125  fix: 删除负载感知描述和死代码    ← 对照审查发现
19cf741  docs: 新增架构演化文档          ← 共同决定 → 初稿 → 多轮修改
```

### 9.3 协作经验

- AI 在代码审查中善于发现模式不匹配和安全漏洞，但理解上下文深度有限
- 架构决策需要多轮讨论，AI 的建议有时需要结合工程直觉做权衡
- 架构演化文档是跨会话传递上下文的可靠方式
- 开发者和 AI 更像"结对讨论"，而非"下命令执行"
